# 交个代（Justify Work）产品说明文档

> 版本：v0.1
> 日期：2026-05-07
> 状态：规划稿

---

## 一、产品定位

**交个代**是一款面向个人用户的桌面端工作管理与 AI 总结助手。

产品主线不是纯工作记录，也不是单纯的待办清单，而是以待办事项为起点，围绕任务执行过程沉淀工作事实，再由 AI 基于这些事实生成日报、周报、月报以及季度、半年度、年度述职草稿。

核心价值体现在三个方面：

1. 帮助用户低成本维护每日待办与执行状态。
2. 帮助用户把任务完成情况快速转化为可信的工作总结。
3. 帮助用户在需要述职时，快速从历史任务中召回成果、案例、问题与成长素材。

---

## 二、目标用户

首发阶段聚焦个人用户，典型用户包括：

- 需要每天维护工作待办并跟进完成情况的职场用户。
- 需要定期输出日报、周报、月报的研发、产品、运营、设计等知识工作者。
- 在季度、半年度、年度节点需要准备述职材料的个人贡献者。

首发阶段不以团队协作、多成员汇总、组织级报表为目标。

---

## 三、核心问题

当前多数用户在工作总结场景中会遇到以下问题：

1. 每天做了很多事，但到写日报或周报时回忆成本高。
2. 待办列表只能看到做没做完，无法自然沉淀关键产出与结果。
3. 周报、月报、述职内容需要重复整理，缺少可复用的长期积累。
4. AI 可以生成文字，但如果缺乏真实任务事实，输出容易空泛失真。

本产品的解决思路是：

用户从待办出发，在任务推进或完成时补充最小必要的结果说明，系统持续沉淀结构化任务数据，AI 再基于这些数据生成可追溯、可编辑的总结内容。

---

## 四、产品原则

### 4.1 待办优先

首页保持“今日任务”作为主入口。用户进入应用后的第一动作应是查看、创建、排序和完成待办，而不是写长篇记录。

### 4.2 记录为辅

记录不是独立主系统，而是附着在任务执行过程中的补充信息。用户可以在推进任务或标记完成时补充一句结果说明、阻塞信息或下一步计划。

### 4.3 AI 先草拟，用户审核

AI 的职责是归纳、提炼、组织结构和润色表达，不直接替代事实采集。所有日报、周报、月报和述职内容默认先生成草稿，再由用户确认。

### 4.4 事实可追溯

每条总结内容都应尽量映射到具体任务、任务更新或结果说明，避免黑盒式生成。

### 4.5 本地优先

首发采用本地优先的数据策略，优先保证隐私、安全和单机可用性，云同步放在后续阶段考虑。

---

## 五、核心使用链路

产品的核心闭环如下：

1. 用户创建今日待办事项。
2. 用户在执行过程中更新任务状态与进展。
3. 用户在任务推进或完成时补充简短结果说明。
4. AI 基于当日任务及更新信息生成日报草稿。
5. 系统基于历史任务与更新信息聚合生成周报、月报。
6. 当用户输入述职大纲后，系统按时间范围与章节要求召回相关任务事实，生成述职草稿。

这个闭环要求任务管理和总结生成共用同一份底层数据，而不是分裂成两个独立系统。

---

## 六、功能范围

### 6.1 首发范围

首发阶段建议纳入以下能力：

1. 今日待办管理。
2. 任务状态管理。
3. 任务进展补充与完成备注。
4. AI 日报生成。
5. AI 周报生成。
6. AI 月报生成。
7. 基于用户自定义大纲的述职素材聚合与草稿生成。
8. 文本、Markdown、可复制内容导出。

### 6.2 暂缓范围

以下能力建议不作为首发前置条件：

1. 多人协作与团队汇总。
2. 云同步与跨设备账号体系。
3. Jira、飞书、企业微信、日历等外部系统深度集成。
4. 首发即支持高自由度 PPT 自动排版。

## 七、核心模块设计

### 7.1 今日任务工作台

这是产品首页与主要工作区域，负责承载每日任务管理流程。

建议包含以下能力：

- 新建任务。
- 查看今日任务列表。
- 按优先级、项目、标签分组或筛选。
- 标记任务状态，如未开始、进行中、已完成、延期。
- 快速补充任务说明、结果和阻塞。

这个模块的设计重点不是复杂项目管理，而是让用户用尽可能低的操作成本维护当天真实工作轨迹。

### 7.2 任务详情与进展补充

单个任务除了标题和状态，还需要承载后续总结所依赖的信息。

建议支持以下补充字段：

- 当前进展。
- 结果说明。
- 关键产出。
- 阻塞问题。
- 下一步计划。
- 相关链接或附件。

其中“结果说明”必须被设计成高频、低门槛动作。用户在完成任务时，最好只需要补一句话即可。

### 7.3 AI 汇总中心

该模块负责基于任务数据生成不同周期的工作总结。

首发需要支持：

1. 日报生成。
2. 周报生成。
3. 月报生成。

每类总结都应支持：

- 一键生成草稿。
- 查看引用了哪些任务或任务更新。
- 手动编辑与确认。
- 再次生成不同风格版本。
- 导出为文本或 Markdown。

### 7.4 述职工作台

该模块服务于季度、半年度、年度述职场景。

核心流程为：

1. 用户输入述职大纲或章节结构。
2. 用户选择时间范围。
3. 系统按章节召回相关任务、任务更新、成果说明和典型案例。
4. AI 基于召回结果生成章节草稿。
5. 用户逐章编辑、补充和确认。

这个模块的重点是“按大纲召回事实”，不是一次性直接生成整篇最终述职。

### 7.5 AI 聊天辅助面板

现有聊天面板可继续保留，但角色需要从“通用聊天助手”收敛为“任务管理与总结助手”。

建议承担以下辅助场景：

- 用自然语言批量创建待办。
- 帮用户总结今天已完成任务。
- 帮用户把零散进展整理成任务更新。
- 帮用户生成特定风格的日报或述职措辞。

聊天面板是增强入口，不应替代任务主流程。

## 八、核心数据模型

### 8.1 Task

任务本体，用于表示用户的待办事项。

建议字段包括：

- id
- title
- description
- project
- priority
- status
- plannedDate
- dueDate
- tags
- estimatedDuration
- actualDuration
- createdAt
- updatedAt

### 8.2 TaskUpdate

任务更新，用于记录任务在执行过程中的阶段信息。

建议字段包括：

- id
- taskId
- updateType
- progressNote
- resultNote
- blockerNote
- nextStep
- attachmentLinks
- createdAt

`TaskUpdate` 是支撑 AI 总结质量的关键对象。没有这层数据，AI 只能根据任务状态做浅层总结。

### 8.3 Summary

周期性总结草稿与定稿。

建议字段包括：

- id
- summaryType
- startDate
- endDate
- title
- content
- sourceTaskIds
- sourceUpdateIds
- draftStatus
- generatedAt
- confirmedAt

### 8.4 ReportDraft

述职草稿对象。

建议字段包括：

- id
- reportType
- outline
- timeRange
- sections
- sourceTaskIds
- sourceUpdateIds
- generatedAt
- updatedAt

## 九、AI 生成逻辑

### 9.1 日报

日报的生成依据主要来自当日任务与任务更新，建议输出以下结构：

1. 今日完成。
2. 关键产出。
3. 未完成事项及原因。
4. 明日计划。

### 9.2 周报

周报的生成依据主要来自一周内的任务推进情况，建议输出以下结构：

1. 本周重点工作。
2. 已完成成果。
3. 推进中的事项。
4. 风险、阻塞与协作需求。
5. 下周计划。

### 9.3 月报

月报需要在周报基础上增加阶段视角，建议输出以下结构：

1. 本月阶段性成果。
2. 重点项目推进情况。
3. 数据、效率或影响变化。
4. 问题与复盘。
5. 下月重点方向。

### 9.4 述职草稿

述职内容不应由 AI 直接自由发挥，而应采用以下流程：

1. 用户输入述职大纲。
2. 系统按章节与时间范围召回相关任务与更新。
3. AI 基于召回事实生成章节草稿。
4. 用户逐章调整表述、补充指标和案例。

## 十、交互重点

### 10.1 完成任务后的最小补充动作

这是整个产品最关键的交互点。

当用户把任务标记为完成时，系统不应强迫用户填写复杂表单，而应该提供一个极轻量的补充入口，例如：

- 你完成了什么？
- 产出了什么结果？
- 是否有后续动作？

如果这一步太重，用户会跳过，AI 总结的质量也会明显下降。

### 10.2 总结内容的可追溯性

当 AI 生成一段日报或周报内容时，用户最好能知道这段内容主要来自哪些任务与更新记录。这样能提升用户对结果的信任感，也便于修正错误归纳。

### 10.3 高价值成果标记

不是所有任务都值得进入述职材料。系统后续应支持用户将某些任务或结果标记为高价值成果，以便在季度或年度阶段优先召回。

---

## 十一、技术实现方向

结合现有 Electron + React + AI Runtime 架构，建议新增以下能力：

1. 在主进程增加结构化本地持久化层，用于保存任务、任务更新、总结和述职草稿。
2. 在共享类型层新增 `Task`、`TaskUpdate`、`Summary`、`ReportDraft` 等模型。
3. 在任务主界面中加入任务更新与完成备注能力。
4. 在 AI 工具层增加任务查询、时间范围聚合和总结生成工具。
5. 在渲染层新增“汇总中心”和“述职工作台”两个功能域。

---

## 十二、阶段规划

### 12.1 第一阶段

目标是跑通最小闭环：

1. 用户创建和维护今日待办。
2. 用户在任务推进或完成时补充简短结果说明。
3. 系统根据当日任务生成日报草稿。
4. 支持周报、月报的基础聚合。

### 12.2 第二阶段

目标是增强长期复用价值：

1. 增加述职工作台。
2. 支持按大纲生成季度、半年度、年度述职草稿。
3. 增加高价值成果标记与检索能力。

### 12.3 第三阶段

目标是扩展导出和外部连接能力：

1. 增加模板化导出。
2. 引入 PPT 模板填充式生成。
3. 评估外部系统导入与云同步。

---

## 十三、验收标准

产品规划是否成功，重点看以下结果：

1. 用户能否在低负担下维护每日待办。
2. 用户完成任务时，是否愿意顺手补充一句结果说明。
3. 系统能否基于任务事实生成可信的日报、周报、月报。
4. 系统能否根据述职大纲快速召回历史任务成果并形成可编辑草稿。
5. 所有生成内容是否具备基本的事实追溯能力。

---

## 十四、结论

这款产品的本质不是“AI 帮你写汇报”，而是“把每日待办和任务执行过程变成可持续积累的工作资产”。

首发阶段最重要的不是做复杂导出，也不是做泛化聊天，而是先把以下一条主链路打透：

待办创建 -> 任务执行 -> 完成补充 -> AI 总结 -> 历史沉淀 -> 述职复用

只要这条链路稳定，日报、周报、月报和述职能力就会形成连续价值，后续再叠加 PPT 导出和外部集成才有意义。
